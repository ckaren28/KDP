## The problem

Museum interpretation is slow, expensive, and under-resourced. A single wall label — 50 to 70 words, written to be understood by a 12-year-old while remaining accurate enough for a specialist — can take several rounds of drafting, editorial review, and curatorial sign-off. Multiply that by a collection of thousands.

At the same time, the stakes are real. Labels make assumptions. They attribute meaning. They decide what to say about provenance, cultural context, and artist intent — and what to leave out. Automation that removes the human editorial step is a different kind of problem.

This tool is built for the middle path: AI-drafted, human-edited.

## How it works

The tool takes artwork metadata (title, artist, year, medium) and a source description, then generates six output types simultaneously:

- **Wall label** (50–70 words) — the standard interpretive label
- **Extended label** (150–200 words) — for more context or complex works
- **Kids label** — accessible language for younger visitors
- **Audio guide script** — written for listening, not reading
- **Alt text** — a functional image description
- **Curator notes** — surfaced assumptions, gaps, and editorial flags

**Variations** generate two alternate wall labels for comparison, making the editorial tradeoffs visible.

**Workshop Mode** adds facilitator prompts and discussion questions, turning the tool into a teaching exercise: generate with *Stay close* ON and OFF, compare the outputs, discuss what changed and why.

## Architecture

```
User input (description + optional metadata)
        ↓
Netlify Function (keeps API key server-side)
        ↓
Anthropic API → JSON { wall_label, extended_label, kids_label,
                       audio_guide_script, alt_text,
                       curator_notes, variations }
        ↓
Copy-enabled output sections
```

The artwork rotator (top of the page) pulls public-domain paintings from the Met Museum Open Access API — a pool of 20 random works, displayed in a crossfade cycle. It updates the wall label card with each artwork's metadata, making the tool immediately contextual without requiring user input.

## Design decisions

**Six outputs in one call.** Running six separate API calls for six label types would be slow and expensive. A single structured prompt produces all six fields as typed JSON keys. The constraint forces the model to hold the same source description in context for all output types simultaneously, which tends to produce more consistent tone across the outputs.

**"Stay close to source text" as a first-class control.** This checkbox changes whether the model can draw inferences beyond the provided description. With it on, the output stays close to what's observable. With it off, the model may interpret, contextualize, or speculate. Neither is always right — the choice depends on what the curator knows and what they want the label to do.

**Curator notes as audit layer.** The model is explicitly prompted to surface its own assumptions in the curator notes field: what it inferred, what's missing from the source text, what might need verification. This is an attempt to keep the human in the loop rather than hiding the model's reasoning inside a clean-looking output.

**Originally built for museums.** The prompt system and output types are equally applicable to fashion archive documentation, garment description, collection notes, and exhibition copy — anywhere interpretive text needs to be drafted at scale with editorial control retained.

## The output system

The design challenge wasn't building an API call. It was defining what 
"accessible" means across six different audiences and encoding those 
distinctions precisely into the prompt structure.

| Format | Reader | Constraint |
|---|---|---|
| Wall label | Standing visitor, 15 seconds | ~80 words, no jargon |
| Extended label | Visitor who pauses | ~200 words, permits depth |
| Kids label | Age 6-10 | Not simplified, differently framed. Questions, wonder, sensory detail |
| Audio guide script | Listener, not reader | Written for the ear. Conversational rhythm, present tense, ~45 seconds spoken |
| Alt text | Screen reader user | Functional replacement for a visual. Objective, WCAG-compliant, no interpretation |
| Curator notes | Internal | Flags ambiguities, suggests research directions, notes what the source description didn't specify |

All six formats are generated in a single API call using structured JSON 
output. Each is independently copyable and regeneratable. Forcing curators 
to regenerate all six when they only want to tweak one is friction that 
kills adoption. The copy-per-section architecture respects how editorial 
workflows actually run.

## Key design decisions

**Structured JSON over streaming text.** All six formats arrive together 
and render into independent copyable cards. Streaming would have made 
per-section copy interactions much more complex, and curators typically 
want all formats at once anyway.

**A single API call for all six formats.** Higher latency per generation, 
but simpler architecture and a better match for how the tool gets used. 
The tradeoff was worth it.

**Tone as a prompt injection, not a separate system prompt.** The tone 
dropdown (Neutral, Warm, Playful, Academic) injects into the user prompt 
rather than switching system prompts. Less precise, but much simpler to 
maintain as a single prompt template.

**The "Show Prompt & Architecture" toggle.** Exposes the system prompt, 
user template, and call flow to anyone who wants to see them. This was 
a deliberate transparency decision. Institutional adopters want to audit 
what the model is doing before they trust it with their collections. 
Showing the work is also a trust signal for visitors who are skeptical 
of AI tools in cultural contexts.

**API key security via Netlify Function.** All calls route through a 
serverless function. The key never touches the browser. Cold starts add 
~300ms latency on first request, which is an acceptable tradeoff for 
a tool used in considered, not rapid-fire, workflows.
